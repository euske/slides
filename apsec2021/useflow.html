1. Background: Consistency is Crucial for Maintaining Large Software Projects

  • Large projects are developed by a team with multiple people.
    • Even if there's only one real developer, they turn into a different person as time passes.
  • Typically, a style guide is set up to facilitate collaboration of team members.
    • Source code style guides are well known:
      • Google JavaScript Style Guide
      • NYTimes Objective-C Style Guide
    • On the other hand, there's little guideline of how to give a name to variable names and function names.
      • Microsoft General Naming Conventions
  • However, identifiers are crucial for program understanding:
    • Programmers rely on the meaning of names. [Lawrie, 06]
    • Bad names obstructs program understanding, resulting in bad code quality. [Avidan, 17]
    • One of the most asked questions by programmers: "Where is this class/method used?" [Sillito, 08]
    • Just searching GitHub Issues with "wrong name" results more than a million results.
    • The importance of names are also emphasized in books, e.g. "Code Complete", "The Practice of Programming".

  • Our research goal was the following:
    • Avoid using a dictionary or heuristics in order to reduce the maintenance cost and cognitive bias.
    • One size doesn't fit all. Aim at a tunable system for each project because the meaning of identifiers varies.
      • "view" in a GUI application.
      • "view" in an SQL engine.
      • Different abbreviation: message and msg, etc.
    • Make the system output explainable by showing how a particular answer was obtained.

3. What is "consistency" anyway?

3.1. General discussion

  • Suppose A → B is expected. Call this knowledge K.
  • Suppose A is satisfied.
  • When B is satisfied, "B is consistent with our knowledge K".
  • When ¬ B is satisfied, "B is not conssitent with our knowledge K".

Example.

  • K ≡ A → B: "Whenever it's cloudy, he carries an umbrella."
  • A: It's cloudy.
  • B: He's carrying an umbrella. (Consistent)

3.2. Program Variables

Focus on the usage of program variables:

  • K ≡ A: "When a variable receives the return value of open() and is passed to the first argument of write()" → B: "The name of variable is out." 
    out = open(...);
write(out, ...);
  • A: "A variable receives the return value of open() and is passed to the first argument of write()" 
    ??? = open(...);
write(???, ...);
  • B: When the variable name ??? is out, then it's consistent with K.

3.3. Make it Bayesian

In reality, most rules do not hold for 100% of the time. Therefore, we introduce the "degree of consistency" by using probablistic rules.

  • There's a high chance of A → B ≡ Probablility P(B | A) is high.
  • When P(B | A) and P(A) are both high, yet P(B) is low → low consistency.

4. Focus on the "use flow" of variables

We define the usage of a variable as a sequence of operations that is applied to the value assigned into that variable. Let us call this dataflow of the variable "use flow".

4.1. Example of use flows

Focus on the variable line in the following snippet: 

private BufferedReader fp;

public String get() {
    String line = fp.readLine();
    int i = line.indexOf(' ');
    return line.substring(0, i);
}

Here is how it is visualized:

The red lines in the above graph show the use flow of the variable line:

  • fp.readLine() → line → #this:indexOf() → #arg1:substring()

This path shows what is assigned to the variable line and how it is used.

4.2. Make It Interprocedural

private BufferedReader fp;

public String get() {
    String line = fp.readLine();
    int i = line.indexOf(' ');
    return line.substring(0, i);
}

public void show() {
    String name = get();
    System.out.println(name+"!!");
}

Note that the value of line in function get() now reaches name in the function show().

Here's the visualization:

Finally, the use flow of the variable line is:

  • fp.readLine() → line → #this:indexOf() → #arg1:substring() → assign:name → L:+ → #arg0:println()
This sequence represents the usage of the variable line in this program.

For the sake of simplicity, we ignore branches in the data flow and treat each path separately.

5. Training and Prediction

We extracted the use flows for every variable in a program (in an interprocedural manner). Then we trained a model that predicts a variable name from a given use flow:

For example, the following use flow predicts ??? = line:

  • fp.readLine() → ??? → #this:indexOf() → #arg1:substring() → assign:name → L:+ → #arg0:println()

If the variable is other than line, its name is regarded not consistent with this usage.

Algorithm:

  • Let V be all the variables.
  • For each v1 ∈ V:
    1. For each v ∈ V except v1 (v ≠ v1), compute P(v.name | v.useflow).
    2. Find the variable name n such that argmax P(name | v1.useflow).
    3. The variable name is consistent if n = v1.name. Otherwise, suggest n as a better name for the variable v1.
  • The actual features includes a set of words: Instead of useflow → outputBufferName, we use:
    • → output
    • → buffer
    • → name
    (Words are segmented by CamelCase or snake_case)

RQ1. useflowは変数名の「使われ方」を表現するのに適切なのか?

まず予備実験として、useflowを本手法に使うことの妥当性を判断する。 もしuseflowが変数名の「使われ方」をうまく表現できていれば、 「同じuseflow」をもつ異なる変数は「同じ使われ方」をしているはずである。

実験1. useflowを使って同じ役割をもつ変数を判定する

各プロジェクトから同じuseflowの類似度が高い (cosine > 0.90) 変数の組を選び、(もとの変数名は隠して) それらが表れているコードを 人間に見せて、同じ変数名にすべきかどうかを判定させる。

  1. 2つの変数名は同じであるべきだ。
  2. 2つの変数名は同じでもよい。
  3. 2つの変数名は違っているべきだ。
  4. わからない。

7. Discussion

本実験の結果から…

  1. useflowは変数の使われ方を表現するのにある程度有効。
  2. 実際のプロジェクトで、39% の確率でもとの開発者よりもよい名前を提案できた。
  3. 各提案で人間に納得できる論拠がきちんと示せたかどうかは証明できなかった。

7.1. Threats to Validity

Internal Validity (実験はRQに答えているか?)

  1. 被験者のプログラミング能力に左右されている。
    → しかし、もし被験者がランダムに選んでいるとしたら Kappa = 0 になるはずなので、被験者9人は ある程度の一般的な基準を有しているといえる。
  2. 被験者が評価コードに対する前提知識があるかもしれない。
    → 今回のケースではない。
  3. Precision のみで Recall が測定できていない。
    → 「完璧なソースコード」を取得することが不可能なので、 Recall は無理。

External Validity (一般化可能か?)

  1. 現状では、プログラミング言語は Java のみ。
  2. 実験プロジェクトおよび被験者の数が充分でない。
  3. useflowの取得方法が完璧でない。 (実装上のミス、機能の欠損)
  4. 動的ディスパッチ、variable aliasing を全部追えていない。
  5. useflow全体を比較せず、エッジを比較するだけの判定では不十分である。
  6. Naive Bayes classifier の性能が不十分である。

8. Conclusion

  • 一貫性を検査するための一般的な枠組みを提案した。
  • 変数の使い方を表現するためのuseflowという形式を提案した。
  • 上の手法を使って、変数名の一貫性を検査・修正することができた。
